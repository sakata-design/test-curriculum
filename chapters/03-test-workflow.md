## 現場でよくある流れ

テスト案件では、次の流れで動くことが多くあります。

1. 仕様や資料を読む
2. 実施対象と観点を確認する
3. テストを実施する
4. エビデンスと結果を残す
5. 問題があれば報告する
6. 必要なら再確認する

## 各工程で意識すること

### 1. 仕様や資料を読む

- 何を確認する案件かを把握する
- 期待結果の基準がどこにあるか確認する
- 画面一覧、フロー図、設計書のうち、どれを参照すべきか確認する

### 2. 実施対象と観点を確認する

- 正常系、異常系、境界値のどこを見るか整理する
- どのシートに記録するか確認する
- エビデンスの保存先と命名ルールを確認する

### 3. テストを実施する

- 手順どおりに確認する
- 感覚ではなく、期待結果と実際の結果の差を見る
- 結果が曖昧なら、その場で資料へ戻る

### 4. エビデンスと結果を残す

- スプレッドシートへ実際の結果と判定を残す
- 必要なスクリーンショットを保存する
- ケース ID とエビデンス名が対応するようにする

### 5. 問題があれば報告する

- 事象、再現手順、期待結果、実際の結果を整理する
- Backlog へ起票する
- 発行された課題番号をシートへ戻す

### 6. 必要なら再確認する

- 修正や回答が入ったら再確認する
- 結果や判定を更新する
- 追跡できるように番号や記録をそろえる

## 課題に入る前に確認するもの

後半の課題に入る前に、少なくとも次を確認してください。

- Android 実機と APK の起動確認
- Drive 上の配布物の場所
- 自分の Drive に用意した作業用テストプログラム
- 記録に使うシート
- エビデンスの保存先
- Backlog の起票先

PR は着手前に作るものではなく、課題の結果がそろったあとに提出時に作成します。
提出時に何をまとめるかは、[結合試験をやってみる](./05-test-execution.md) と [単体試験を設計してみる](./06-test-design.md) で確認してください。

## よくあるミス

- 作業を始めてから必要資料を探す
- 共有 Drive 側を直接編集する
- エビデンスの保存先や命名ルールを確認せずに進める
- 問題が出てから報告に必要な項目を確認し始めて、再現手順や期待結果を取りこぼす
