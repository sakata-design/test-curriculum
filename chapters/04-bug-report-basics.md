# 第6章: 再現できるバグ報告の書き方

## この章のゴール

- バグ報告に最低限必要な要素を説明できる
- 感想ではなく事実で書く意識を持てる
- 次章の結合試験課題で、起票に必要な項目を迷わず整理できる

## バグ報告に必要な5要素

1. 何をしたか
2. 何が起きたか
3. 本来どうあるべきか
4. 再現手順
5. 環境情報

## 伝わる書き方

- 「変だった」ではなく、何がどう違ったかを書く
- 画面名、操作、入力値を省略しない
- 期待結果と実際の結果を並べて書く

## このプログラムで使うテンプレート

後半の課題では、主に次の2つを使います。

- 下書き用: `templates/bug-report-template.md`
- 実運用: Backlog

テンプレートは任意の下書き用です。まずは見出しの意味だけ理解し、実際に起票したらシート側へ起票番号を戻す流れを押さえてください。
このプログラムで使う起票先は [TESTCURRICULUM プロジェクト](https://convi0310.backlog.com/projects/TESTCURRICULUM) です。
ログインができない場合、ログイン後 TESTCURRICULUM のプロジェクトにたどり着けない場合は、その時点で講師へすぐ連絡してください。
起票の仕方は、[Backlog Wiki のチケット起票ページ](https://convi0310.backlog.com/wiki/TESTCURRICULUM/%E3%83%81%E3%82%B1%E3%83%83%E3%83%88%E8%B5%B7%E7%A5%A8) を参考にしてください。

Backlog での実際の流れは次の通りです。

1. TESTCURRICULUM プロジェクトを開く
2. 新しい課題を追加する
3. 課題種別を `バグ` にする
4. 件名と本文を入力する
5. 保存したら、発行された課題番号をスプレッドシートへ戻す

課題番号は、たとえば `TESTCURRICULUM-123` のような形で表示されます。

Backlog を使う前に、次を確認してください。

- どの課題種別を使うか
- 優先度をどう付けるか
- 自分で担当を付けるか

このプログラムでは、次のルールで揃えます。

- 課題種別は `バグ`
- 件名は `[結合テスト][ケースID][画面名] 事象`
- 優先度は「進行不能なら高め、回避可能なら中以下」を目安にする
- 担当者は講師指示がなければ空欄でよい

```text
事象:
再現手順:
期待結果:
実際の結果:
補足:
```

起票後にやることも忘れないでください。

- 保存後に発行された課題番号を確認する
- 実施シートの `備考` 列へその番号を記載する
- 必要ならエビデンス名も一緒に残して、あとで照合できるようにする

## よくあるミス

- 感想や推測だけを書いて、事実が抜ける
- 再現手順が「同じことをしたら起きる」になっている
- 期待結果が書かれていない
- 件名にケース ID や画面名がなく、あとで追えない
- 起票しただけで満足し、実施シートへ番号を書き戻していない

## AIに聞いてみよう

```text
目的：バグ報告を短く分かりやすくしたい
現状：事象と手順は書けるが、期待結果が曖昧になりがち
詰まりポイント：どこまで書けば再現しやすいか分からない
自分の仮説：画面名、入力値、押したボタンを明記するとよさそう
確認したいこと：この報告文で足りない情報を教えて
```
