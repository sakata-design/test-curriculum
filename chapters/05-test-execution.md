# 第7章: 結合試験をやってみる

> 提出ブランチ：`feature/05-test-execution`（PRのbase：`develop`）

## この章のゴール

- ストアからインストールした対象アプリを使って結合試験を実施できる
- 実施結果をスプレッドシートとエビデンスで残せる
- テスト結果を PR で提出する流れまで通しで体験できる

## この章の位置づけ

この章は、課題1 の具体的な実施手順をそのまま進めるための章です。
[テスト案件の基本フロー](./03-test-workflow.md) では現場でよくある流れを整理しましたが、この章では「今、自分が何をやるか」に絞ります。
[再現できるバグ報告の書き方](./04-bug-report-basics.md) は、起票内容に迷ったときに見返してください。

## 事前準備

この章では、Drive 上の課題1資材と、ストアからインストールした対象アプリ「やりくり」を使います。
着手前に次をそろえてください。

- 対象アプリ「やりくり」のインストールと起動・バージョン確認（→ [実施環境を準備する](./01-setup.md)）
- テストに使うスマホ（iPhone または Android）
- 自分の Drive にコピーした `テストプログラム_<氏名>_YYYYMMDD > 結合テスト > テストケース`
- 自分の Drive にコピーした `テストプログラム_<氏名>_YYYYMMDD > 結合テスト` 内の設計書（画面遷移図・画面設計書・詳細設計書）
- 起票先の [TESTCURRICULUM プロジェクト](https://convi0310.backlog.com/projects/TESTCURRICULUM)
- 自分の Drive にコピーした `テストプログラム_<氏名>_YYYYMMDD > 結合テスト > エビデンス`

ログインができない場合、ログイン後 TESTCURRICULUM のプロジェクトにたどり着けない場合は、この章を始める前に講師へすぐ連絡してください。
テストに使うスマホ（iPhone または Android）が用意できない場合も、この章を始める前に講師へすぐ連絡してください。
まだ自分の Drive に作業用コピーを作っていない場合は、先に [教材・Drive・Backlogの使い分け](./02-assets-and-tools.md) に戻って準備してください。
PR はこの時点ではまだ不要です。実施結果、シート URL、必要なエビデンスがそろったあとに提出用として作成します。

## この章で提出までにやること

1. 自分の Drive にある作業用テストプログラムを開く
2. 画面遷移図・画面設計書・詳細設計書を見ながら観点を理解する
3. 実際に画面を操作して結果をシートへ記録する
4. 問題があれば Backlog へ起票し、シートへ戻す
5. シート URL と結果を PR（提出内容をレビューしてもらうためのページ）で提出する

## 手順

### 1. 作業用のテストプログラムを開く

共有 Drive を直接編集せず、[教材・Drive・Backlogの使い分け](./02-assets-and-tools.md) で作成した自分の Drive の `テストプログラム_<氏名>_YYYYMMDD` を開きます。
課題1 では、その中の `結合テスト` をそのまま使うので、`テストケース` フォルダを追加でコピーし直す必要はありません。

`結合テスト > テストケース` を開き、先頭の `サマリー` シートと機能別シート（支出／履歴／予算／貯蓄目標／貯蓄プール／設定／バックアップ／リワード広告／E2E一気通貫）がそろっていることを確認します。機能別シートを課題1 の作業用シートとして使い、進捗は `サマリー` で確認します。
あわせて、`結合テスト > エビデンス` フォルダがあることを確認し、その既存フォルダをそのまま使ってください。

`結合テスト` フォルダまたは配下のシートが見当たらない場合は、この章を開始せず講師へすぐ連絡してください。

講師から別指示がない限り、課題1 では機能別シートを順に実施し、`サマリー` で全体の進捗を確認します。

### 2. 参照資料を開く

少なくとも次を開いた状態で進めます。

- `テストケース`（`サマリー` ＋ 機能別シート）
- 設計書（画面遷移図・画面設計書・詳細設計書）

### 3. 実機（iPhone / Android）でテストを実施する

シートの項目を見ながら、少なくとも次を確認します。

- 正常系で基本フローが通るか
- 異常系で明らかに壊れないか
- 画面遷移やメッセージが設計書と大きくずれていないか

### 4. 結果とエビデンスを残す

- 実施したケースごとにシートへ結果を残す
- スクリーンショットは、自分の Drive にコピーした `テストプログラム_<氏名>_YYYYMMDD > 結合テスト > エビデンス` に、ケース ID と対応する名前で保存する
- バグがあれば Backlog へ起票し、シートの備考へ起票番号を記載する
- 必要に応じて `templates/evidence-log-template.md` を個人メモとして使う

`エビデンス` は、テスト中に取得したアプリのスクリーンショットなどを残すためのものです。
この課題では、コピーした `テストプログラム_<氏名>_YYYYMMDD > 結合テスト` の中にある `エビデンス` に統一します。

### 5. Backlog 起票が必要ならシートへ戻す

不具合があった場合は、Backlog へ起票して終わりにしません。
保存後に発行された課題番号を、必ず該当する機能別シートの `備考` 列へ戻します。
起票内容に迷った場合は [再現できるバグ報告の書き方](./04-bug-report-basics.md) を見返してください。
起票画面の入力方法は、[Backlog Wiki のチケット起票ページ](https://convi0310.backlog.com/wiki/TESTCURRICULUM/%E3%83%81%E3%82%B1%E3%83%83%E3%83%88%E8%B5%B7%E7%A5%A8) を参考にしてください。

### 6. 完了条件を確認する

課題1 は、次の状態になったら完了です。

- テストケースの対象範囲を実施している
- 自分の Drive 側にある `テストケース` のシート URL を提出できる状態になっている
- シートに判定が入っている
- 必要なエビデンスが保存されている
- 必要に応じて Backlog 番号がシートの備考へ戻っている
- PR に提出情報をまとめている

PR には、少なくとも次をまとめてください。

- 自分の Drive 側にある `テストケース` のシート URL
- コピーした `テストプログラム_<氏名>_YYYYMMDD > 結合テスト > エビデンス` の URL
- 必要に応じて Backlog 番号

## よくあるミス

- 結果だけ書いて、どの手順で確認したか残していない
- 共有 Drive 側を直接開いたまま作業している
- スクリーンショットの名前とケース ID が対応していない
- Backlog の起票先を確認しないまま実施を始めている
- 一部の機能別シートだけで終えて、他の機能別シートを見落としている
- 問題が出たのに、起票番号をシートの備考へ戻していない

## AIに聞いてみよう

```text
目的：結合試験の結果整理をしたい
現状：正常系は確認できたが、異常系の記録が曖昧
詰まりポイント：どこまで書けば他の人が追えるか不安
自分の仮説：手順、期待結果、実際の結果をセットで書けば伝わる
確認したいこと：この記録で不足している観点を教えて
```
